# cc-sessions-coord — Design-Spec

## 1. Problem

Zwei Systeme leiden unter ephemerer Sessions-Schicht als Persistenz-Schreiber:

- **Memory-Sync** (Memory-Files → Hindsight/Shodh): Hook→Claude→MCP — Claude vergisst Calls, Compact skipt, Sessions crashen → 23 stale Claims.
- **Coord** (Cross-Session-Inject, Activity-Tracking): mehrere parallele Sessions schreiben gleichzeitig in lokale SQLite (`~/.claude/sessions-coord.db`) → unter Last "database is locked"-Hänger, fragile Multi-Session-Races.

Beide brauchen denselben Fix: **Single-Writer-Backend hinter HTTP-API.**

## 2. Architektur

```
[Claude-Sessions]  (viele parallel)
    │
    │ HTTP POST/GET (auf localhost:7733)
    ▼
[NSSM-Service: cc-sessions-coord]
  ├── Kestrel + Minimal API     (Endpoints für Sync + Coord + Health)
  ├── Npgsql                    (single writer in cc_sessions_coord-DB)
  └── BackgroundService (30s)   (pusht Memory-Queue an Hindsight + Shodh)
    │
    ▼
[Postgres cc_sessions_coord]    ── lokal auf 5432
  ├── schema coord              (sessions, activities, injections, injection_logs)
  └── schema memory_sync        (queue, dlq, worker_state)
```

**Single Writer:** Nur der Worker schreibt in die DB. Sessions schreiben NIE direkt — alles via HTTP.

**Stack:** .NET 8 Worker (`Microsoft.NET.Sdk.Web`), Kestrel, Minimal API, Npgsql 10.0.2, Swashbuckle/OpenAPI für Dev-Doku, xUnit + WireMock.Net für Tests.

## 3. Datenbank — `cc_sessions_coord`

**Postgres-Datenbank** in der bestehenden Postgres-18-Instanz auf 5432. Eigene DB, kein Schema-Mix mit VibeHub. Bereits angelegt, kanonische DDL: `migrations/V001__initial.sql` (siehe Repo).

**Schemas:**
- `coord` — Sessions, Activities, Injections, Injection-Logs (1:1 portiert aus existing SQLite-DB `~/.claude/sessions-coord.db`)
- `memory_sync` — Queue mit Per-Sink-State, DLQ, Worker-State

**Tabellen-Skizze (Details in `migrations/V001__initial.sql`):**
```sql
-- coord-Schema (aus SQLite portiert, gleiche Spalten)
coord.sessions     (session_id PK, pid, project_path, ..., last_heartbeat, status, schema_version)
coord.activities   (id PK, session_id FK, action, target_path, started_at, ended_at, schema_version, ...)
coord.injections   (id PK, source_session_id, target_session_id, inject_text,
                    priority, created_at, delivered_at, acknowledged_at, schema_version, ...)
coord.injection_logs (id PK, injection_id FK, event, ts)

-- memory_sync-Schema
memory_sync.worker_state (id=1 PK, worker_epoch UUID, fencing_token BIGINT, started_at, last_heartbeat)
memory_sync.fencing_seq  (Postgres-Sequence)
memory_sync.queue (
    event_id          UUID PK,
    schema_version    INT DEFAULT 1,
    source            TEXT,                -- 'hook' | später 'jsonl-tail'
    session_id        TEXT,
    message_id        TEXT,
    memory_path       TEXT,
    memory_type       TEXT,
    memory_name       TEXT,
    content           TEXT,
    payload           JSONB,
    created_at        TIMESTAMPTZ,
    locked_by         UUID,
    locked_until      TIMESTAMPTZ,
    fencing_token     BIGINT,
    hindsight_status  TEXT DEFAULT 'pending',   -- pending|ok|error_permanent|error_temporary
    hindsight_attempts INT DEFAULT 0,
    hindsight_last_attempt_at TIMESTAMPTZ,
    hindsight_last_error TEXT,
    hindsight_next_retry_at TIMESTAMPTZ,
    shodh_status      TEXT DEFAULT 'pending',
    shodh_attempts    INT DEFAULT 0,
    shodh_last_attempt_at TIMESTAMPTZ,
    shodh_last_error  TEXT,
    shodh_next_retry_at TIMESTAMPTZ,
    completed_at      TIMESTAMPTZ
)
memory_sync.dlq   (event_id PK, payload jsonb, reason TEXT, classified_error TEXT, dlq_at TIMESTAMPTZ)
```

**Dedup-Key `event_id`** (Council-Verfeinerung): UUIDv5 aus `(source || session_id || message_id || memory_path || sha256(content))`. NICHT nur content-Hash — sonst werden legitime Wiederholungen derselben Memory geschluckt. `message_id` aus Hook-Input (CC-Tool-Call-Identifier), eindeutig pro Edit. Worker-`INSERT ... ON CONFLICT (event_id) DO NOTHING` macht Hook-Retries idempotent ohne legitime Updates zu fressen.

**Schema-Versioning** (Council): jeder Sync-Event-Payload hat `schema_version: 1` plus zentralen `payload jsonb`-Block für additive Felder. Expand-contract-Migrationen: neue Felder in `payload` ergänzen, Worker liest beide Versionen, alte Felder erst entfernen wenn Hook-Pool durch ist. Coord-Tabellen analog mit `schema_version`-Spalte.

**Fencing-Token monoton** (Mistral Round 2): zusätzlich zum UUID-`worker_epoch` ein `fencing_token bigint` aus Postgres-Sequence (`memory_sync.fencing_seq`). Beim Worker-Start: `nextval()` → in `worker_state`-Tabelle persistieren. Lease-Update prüft `fencing_token >= current`, alter halb-toter Worker mit niedrigerem Token kann nicht überschreiben. Verhindert Split-Brain auch bei kompromittierter Uhr/UUID-Kollision.

**Sink-Idempotenz live verifiziert** (siehe `docs/sink-idempotency-2026-05-07.md`):
1. **Hindsight** (MCP-JSON-RPC `tools/call retain`): `document_id` ist serverseitiger Idempotenz-Anker. Zwei identische POSTs → ein einziges Memory mit `chunk_id = default_<document_id>_0`.
2. **Shodh** (REST `POST /api/remember`): Server dedupliziert automatisch auf content-Hash pro `user_id`. Zwei identische POSTs → identische `id` zurück.

Konsequenz: Reconciliation-Loop ist **kein Pflicht-BackgroundService mehr**, sondern optionales Phase-2-Safety-Net.

## 4. Worker — HTTP-Endpoints

**Listener auf `localhost:7733`** (ASP.NET Core Minimal API auf Kestrel, gebunden via `WebApplication.CreateBuilder`-Configuration `urls=http://localhost:7733`). Kestrel hört nur auf 127.0.0.1, NICHT `0.0.0.0` — physisch nur erreichbar von Prozessen auf derselben Maschine.

**Auth-Modell: Loopback-Trust.** Kein API-Key, kein Bearer-Token zwischen Hook und Worker. Sicherheitsannahme: wer 127.0.0.1 erreicht, hat eh Code-Execution auf dieser Maschine — Auth wäre rein dekorativ. Optional doppelte Absicherung: in einem Endpoint-Filter `HttpContext.Connection.RemoteIpAddress` gegen `IPAddress.Loopback`/`IPAddress.IPv6Loopback` validieren (Belt+Suspender). Sessions identifizieren sich über `session_id` im Payload, der wird aber NICHT kryptografisch verifiziert — pure Tracking-Information.

| Method | Path | Zweck |
|---|---|---|
| GET | `/health` | erweitert: queue_length, queue_lag_s (älteste pending - now), processing_length, dlq_length, dlq_growth_24h, last_sync_at, last_dlq_at, sink_error_rate_24h, postgres_deadlock_count_24h, worker_epoch, uptime_s, stuck_events_count |
| POST | `/sync` | Memory-Sync-Event entgegennehmen, in `memory_sync.queue` ablegen, 202 zurück |
| POST | `/coord/session` | Session registrieren (heartbeat, init) |
| POST | `/coord/activity` | Activity start/end |
| POST | `/coord/injection` | Cross-Session-Inject senden |
| GET | `/coord/injections?for=<sid>` | Pending-Injections für Session abholen |
| POST | `/coord/injection/{id}/ack` | Inject quittieren |
| GET | `/coord/sessions` | Aktive Sessions auflisten |

**Hook-Timeout** beim POST: **1000ms** mit Client-seitigem **Circuit-Breaker** (3 Failures in 60s → 30s open, dann half-open Probe). Worker schreibt sofort durable in DB (Postgres-WAL), antwortet **202 Accepted** ohne auf Sink-Sync zu warten. Falls Worker-down: Hook logged silent in `~/.claude/logs/hook-failures.log` und exitet 0 — Memory-Edit darf NIE blockiert werden.

**Begründung 1000ms (Council):** 150ms zu kurz für "HTTP + Postgres durable INSERT" unter AV-Scan, NSSM-Restart, kalter DB-Connection. Stressfall droppt sonst lautlos.

### 4a. Sink-Clients

Im Worker leben zwei dedizierte Client-Klassen, die pro Sink-Aufruf eine eigene HTTP-Connection aufmachen. Beide werden über `IHttpClientFactory` (DI-registriert) abgerufen, damit Connection-Pooling und HttpClient-Disposal sauber laufen.

**Hindsight (`HindsightClient`):**
- Endpoint: `http://127.0.0.1:8888/mcp/default`
- Transport: MCP-JSON-RPC über Streamable-HTTP. Worker baut bei jedem Sync-Cycle eine `tools/call`-Anfrage mit Tool `retain`.
- Stateless-Mode reicht: jeder POST eine eigene HTTP-Connection, kein `Mcp-Session-Id`-Header nötig (also kein `initialize`-Handshake).
- Antwort kommt als Server-Sent-Events-Stream (`event: message\ndata: {…JSON…}\n\n`); Client liest den Body bis zum doppelten `\n\n`, parst die `data:`-Zeile als JSON-RPC-Result.
- **Idempotenz-Anker:** `document_id = event_id` (UUIDv5 als Hex-String). Hindsight-Server legt damit `chunk_id = default_<document_id>_0` an und dedupliziert serverseitig.
- Pflichtfelder: `content`. Wir setzen zusätzlich `context = "cc-sessions-coord"`, `document_id = event_id`, `tags = [memory_type]`, `metadata = { source, session_id, memory_path }`.
- Antwort `accepted` ist asynchron — das tatsächliche Memory ist erst ~4 s später in `list_memories` sichtbar. Für unseren `fire-and-mark-ok`-Loop unkritisch.

**Shodh (`ShodhClient`):**
- Endpoint: `http://127.0.0.1:3030/api/remember`
- Auth-Header: `X-API-Key: <SET_VIA_ENV>` (nicht `Authorization: Bearer`).
- Pflichtfelder: `user_id` (string) + `content` (mindestens 10 Zeichen, kein Em-Dash/U+2014 — Parser-Bug, ASCII-Strich verwenden).
- Antwort `{"id": "<uuid>", "success": true}`.
- Server dedupliziert automatisch auf content-Hash pro `user_id`. Zusätzliche `event_id` als Idempotenz-Key ist nicht nötig — wir senden sie trotzdem im Body als Tracking-Information.
- Wir setzen `user_id = "cc-sessions-coord-{machine}"` (z.B. `cc-sessions-coord-<YOUR-HOST>`), `content` = Memory-Inhalt.

**`ErrorClassifier`** (gemeinsam genutzt): mappt HTTP-Status + Exception auf `SinkStatus`:
- `2xx` → `Ok`
- `4xx` außer `408` und `429` → `ErrorPermanent` (direkt DLQ-Kandidat)
- `5xx`, `408`, `429`, `TaskCanceledException`/Timeout, Connect-Fail → `ErrorTemporary` (Retry mit Backoff)

## 5. Sink-Sync-Loop (BackgroundService) — mit Lease + Worker-Epoch

Im Worker-Prozess parallel zum Kestrel-Listener läuft ein `BackgroundService` (`SinkSyncService : IHostedService`), der alle 30 s einen Cycle ausführt. Pseudocode (DB-Logik):

```sql
-- beim Worker-Start (SingleInstanceLockService.StartAsync):
SELECT pg_try_advisory_lock(:WORKER_LOCK_ID);  -- single-instance-Schutz
-- WorkerStateRepository.AcquireLease():
SELECT nextval('memory_sync.fencing_seq');
INSERT INTO memory_sync.worker_state (id, worker_epoch, fencing_token, started_at, last_heartbeat)
VALUES (1, :epoch, :token, now(), now())
ON CONFLICT (id) DO UPDATE
   SET worker_epoch = :epoch, fencing_token = :token, started_at = now(), last_heartbeat = now();

-- alle 30 s im SinkSyncService.ExecuteAsync():
-- Lease-Claim atomar (verhindert Doppel-Verarbeitung bei Split-Brain):
UPDATE memory_sync.queue
   SET locked_until  = now() + interval '5 minutes',
       locked_by     = :worker_epoch,
       fencing_token = :fencing_token
 WHERE event_id IN (
   SELECT event_id FROM memory_sync.queue
    WHERE completed_at IS NULL
      AND (hindsight_status IN ('pending','error_temporary')
           OR shodh_status IN ('pending','error_temporary'))
      AND (locked_until IS NULL OR locked_until < now())
      AND (hindsight_next_retry_at IS NULL OR hindsight_next_retry_at <= now() OR hindsight_status='ok')
      AND (shodh_next_retry_at     IS NULL OR shodh_next_retry_at     <= now() OR shodh_status='ok')
    ORDER BY created_at LIMIT 50
    FOR UPDATE SKIP LOCKED
   )
 RETURNING *;

-- für jede claimed row:
--   für jeden Sink mit status != 'ok':
--     resp = await client.SendAsync(row)  -- 15s timeout
--     classify(resp):
--       Ok:               UPDATE ... <sink>_status='ok'
--       ErrorPermanent:   UPDATE ... <sink>_status='error_permanent'  -- direkt DLQ-Kandidat
--       ErrorTemporary:   UPDATE ... attempts+1, next_retry_at = exp-backoff(5/25/125 min)
--   wenn beide Sinks ok:                        UPDATE completed_at = now()
--   wenn ein Sink permanent ODER attempts>=3:   INSERT INTO dlq, DELETE FROM queue
--   sonst:                                      UPDATE locked_until = NULL  (nächster Cycle)

-- ReaperService alle 60 s:
UPDATE memory_sync.queue
   SET locked_until = NULL, locked_by = NULL
 WHERE locked_until < now();
```

**Fenced Worker-Epoch** (Council): jede Worker-Instanz hat eigene Epoch. Halb-toter alter Worker dessen `locked_by != current_epoch` darf nicht committen — UPDATE-WHERE-Clause prüft Epoch und Fencing-Token.

**Fehler-Klassifizierung** (Council, jetzt in `ErrorClassifier`): 4xx (Schema-Verstoß, Auth-Failure) ist *permanent* → direkt DLQ mit reason. 5xx/Timeout/408/429 ist *temporär* → Retry mit Backoff.

**Backoff:** 5 min / 25 min / 125 min, dann DLQ.

**Reconciliation-Loop (optional, Phase 2):** Da beide Sinks live als idempotent verifiziert sind, ist ein Reconciliation-`BackgroundService` (5 min, GET-prüfen ob `event_id` schon im Sink) **kein Pflicht-Bestandteil** mehr. Falls in Phase 2 trotzdem gewünscht (für stuck-event-Recovery), separater `ReconcileService` mit Hindsight `list_memories`-Query und Shodh-Recall-Endpoint. (TODO: Shodh-Lookup-API für `event_id` muss noch verifiziert werden — `/api/remember` selbst ist write-only.)

## 7. Hook-Cutover

**Auf HTTP umstellen:**
- `memory-sync-watch.sh` (PostToolUse) → `curl -X POST localhost:7733/sync -m 1.0` (Bash bleibt Bash, ruft den .NET-Worker via HTTP)
- `session-coord/*.ps1` (Sessions, Activities, Injections) → alle SQLite-Calls durch `Invoke-RestMethod` auf `/coord/*` ersetzen
- `coord-subcommands.ps1` (User-CLI für `/coord status` etc.) → ruft HTTP-Endpoints
- `memory-sync-queue-healthcheck.sh` → pollt `/health`, warnt bei queue>100 oder dlq>0

**Löschen:**
- `memory-sync-flush.sh` (UserPromptSubmit) — obsolet
- `flush-queue.ps1` — obsolet
- File-Queue-Verzeichnisse — obsolet

Wichtig: Hook-Layer und Service-Setup bleiben Shell/PowerShell — nur der Worker selbst ist .NET. Hooks rufen den Worker ausschließlich per HTTP, also ist die Implementierungssprache des Workers für sie irrelevant.

## 9. Failure-Modes (Council-Findings, Stand 2026-05-07)

| Mode | Mitigation |
|---|---|
| Worker down | Hook 1000ms-timeout + Circuit-Breaker + silent fail → Memory-Edit nicht blockiert |
| Hook-Replay (Edit triggert mehrfach) | UUIDv5(`source+session_id+message_id+path+sha256(content)`) + `INSERT ... ON CONFLICT DO NOTHING` |
| Sink Hindsight ok + Shodh fail | Per-Sink-Status → nur Shodh retried |
| Postgres down | HTTP-Endpoint 503, Hook silent, Healthcheck warnt |
| Worker-Crash mid-loop | Lease (`locked_until`) + Reaper-Loop (60s) gibt Eintrag wieder frei nach 5min |
| Split-Brain (alter Worker halb-tot, neuer übernimmt) | Worker-Epoch (UUID pro Start) + UPDATE-WHERE `locked_by = epoch` → alter Worker verliert Lease beim Commit |
| Permanente 4xx Sink-Errors | `ErrorClassifier` → 4xx (außer 408/429) direkt DLQ mit `reason='permanent_4xx'`, kein endloser Retry |
| Temporäre 5xx Sink-Errors | Backoff 5/25/125min, nach 3 fails DLQ mit `reason='exhausted_retries'` |
| DLQ wächst unbemerkt | `/health` exposes `dlq_growth_24h` + `sink_error_rate_24h`; Healthcheck-Hook warnt bei dlq>0 oder rate>5% |
| Schema-Drift Hook↔Worker↔DB | `schema_version` im Payload + `payload jsonb` für additive Felder; expand-contract-Migrationen |
| Auth-Token-Rotation Sinks | Env-Reload via Service-Restart (acceptable downtime <5s) — alternativ `IOptionsMonitor`-File-Watch in Phase 2 |
| Postgres-Deadlock | `/health` exposes `postgres_deadlock_count_24h`; Healthcheck-Hook warnt |
| WAL-Retention für PITR | NSSM-Setup-Schritt: `wal_keep_size = 1GB` und `cc_sessions_coord` in pg_basebackup-Schedule aufnehmen |
| Hook Circuit-Breaker offen | Nach 3 Fails in 60s → 30s open, Hook ist no-op (Memory-Edit trotzdem persistent auf Disk) |
| Sink-Drift bei Timeout-Retry (Sink hat persisted, Client sah Timeout) | Beide Sinks live als idempotent verifiziert (siehe `docs/sink-idempotency-2026-05-07.md`) — Worker-Retry liefert keinen Duplikat. Reconciliation-Loop ist nur optionales Phase-2-Net. |
| Network-Split mit altem halb-toten Worker | Fencing-Token (Postgres-Sequence) monoton steigend + CAS-Update (`WHERE fencing_token <= :current`); alter Worker bekommt SQL-NoOp |
| Stuck Events (lokal commited, remote nie bestätigt) | `/health` exposes `stuck_events_count` (>1h pending, kein DLQ) → Alert |
| HA / Postgres-Single-Point-of-Failure | **Bewusst out-of-scope.** Solo-Dev-Workstation, NSSM-Restart fängt 99% ab. HA-Cluster (Patroni+ETCD) wäre Kanonen auf Spatzen — wenige Minuten Coord-Visibility-Lücke akzeptiert, kein Datenverlust. |
