## 1. Channels-Protokoll — wie es wirklich laeuft (Stand offizielle Doku)

Quelle: https://code.claude.com/docs/en/channels-reference

- **Bridge = MCP-Subprocess**: `claude.exe` spawnt die Bridge per stdio bei Start. Standard-MCP-Lifecycle.
- **Capability**: `capabilities.experimental['claude/channel'] = {}` registriert den Notification-Listener.
- **Notification-Method**: `notifications/claude/channel` mit `{ content: string, meta?: Record<string,string> }`.
- **Render-Verhalten**: jede eingehende Notification wird **inline in den Context der laufenden Session** als `<channel source="..." attr="val">content</channel>` Tag injiziert.
- **Kein Sub-Agent-Spawn auf Receiver-Seite** laut Doku — der Tag landet im Context der bestehenden Session, kein Task-Tool-Aufruf, kein Modell-Switch.
- **Coalescing**: "If several notifications arrive while Claude is busy, they're delivered together on the next turn." Queue + Batch.
- **`instructions`-String** aus dem `Server`-Constructor wird einmalig in den System-Prompt der Empfaenger-Session aufgenommen — nicht pro Notification.
- **Reply-Tool**: Bridge exposed Standard-MCP-Tool, Claude im Receiver ruft es ueber den normalen Tool-Use-Pfad.
- **Permission-Relay**: Optional via `claude/channel/permission`-Capability. Nur fuer Tool-Approvals (Bash/Edit/Write), nicht fuer Trust-Dialoge.

## 2. Pruefung der Warmup-Sub-Agent-Theorie

**Hypothese vom User**: CC spawnt bei jeder Channel-Notification einen "Warmup"-Sub-Agent (agent-*.jsonl) der das Parent-Modell (Opus) erbt → ~4K Tokens pro Klopfer.

**Stand Doku**: Keine Erwaehnung eines Warmup-Mechanismus. Channels schicken nur Tags in den bestehenden Context.

**Empirisch zu pruefen** (laufen jetzt im Hintergrund — siehe Abschnitt 8):
1. Strings im `claude.exe`-Bundle nach `warmup|sidechain|channel.*spawn` durchsuchen
2. CC Node-Bundle (falls auf Windows vorhanden) nach `Warmup|isSidechain|channel.*subagent` greppen
3. Hooks in `~/.claude/hooks/` nach Channel-Event-Triggern absuchen
4. Repo-Scripts auf irgendwelche eigenen Sub-Agent-Spawns nach Channel-Empfang absuchen

Falls die Theorie stimmt: Warmup ist Closed-Source-Verhalten von `claude.exe`. Workaround: parent-Sessions mit Haiku starten statt Opus, oder Channel-Notifications drosseln (Rate-Limit in Bridge).

Falls die Theorie nicht stimmt: Token-Burn ist Context-Wachstum durch Tags + doppelte Notifications (siehe 3).

## 5. cc-yolo-Wrapper — Verhaltens-Regeln (heute finalisiert)

- `cc-yolo <Name>`: neue Session, setzt `$env:CCSC_SESSION_NAME` UND uebergibt `--name <Name>` an claude
- `cc-yolo` (ohne Arg): Read-Host fuer Name, dann wie oben
- `cc-yolo -r <Name>`: Resume per Search-Term, setzt `$env:CCSC_SESSION_NAME` (damit Bridge die richtige Session findet), uebergibt `--resume <Name>`, KEIN `--name`
- `cc-yolo -r` ohne Term: **verboten** — cc-yolo weiss nicht welche Session geladen wird, alter ENV-Name wuerde stillschweigend an die falsche Session kleben
- `cc-yolo -c`: claude-Standard-Continue, KEIN `--name`, KEIN ENV-Touch
- `--resume`/`--continue` (lange Form): manuell aus `$NameOrArgs` rausgezogen, weil PowerShell nur `-Param` bindet
- cwd-Encoding: `[^A-Za-z0-9]` → `-` (Claude Code ersetzt auch Punkte)
- Echo-Format: `[cc-yolo] $env:CCSC_SESSION_NAME='X'; claude --dangerously-skip-permissions --dangerously-load-development-channels server:ccsc-channel --name 'X'` — kopierbar
- Hostname-Guard im OneDrive-Profile: `if ($env:COMPUTERNAME -eq '<YOUR-HOST>')` plus `Test-Path`
- Profile-Snippet `scripts/profile-snippet.ps1` als Dot-Source-Einbindung, damit das Profile nicht driftet

## 8a. Auswertung der Warmup-Sub-Agent-Theorie (empirisch, 2026-05-14 18:00)

Suchmuster gegen 228 MB `claude.exe` (Bun-bundled) und 11 MB `cli.js` (npm-Cache, lesbare JS-Quelle):

**Treffer in `cli.js` (Beweise):**

1. **`warmup`** existiert als Goal-Category fuer Sub-Agents:
   - Schema-Description: `"Maximum number of agentic turns (API round-trips) before stopping. Used internally for warmup."`
   - Goal-Categories-Map: `..., warmup_minimal: "Cache Warmup", ...`
   - Onboarding-Content-ID: `new-user-warmup` (Start-Tipps fuer neue User — irrelevant)
2. **`isSidechain`** ist Flag in JSONL-Entries; markiert Sub-Agent-Messages vs. Haupt-Session-Messages.
3. **`sidechain transcript`** wird per `V31(...)` geschrieben (`Failed to record sidechain transcript`).
4. **`agent-*.jsonl`** ist das File-Pattern fuer Sub-Agent-Transcripts (Funktion filtert JSONLs mit `.startsWith("agent-")`).

**Was nicht im Code zu finden war:**
- Kein Codepfad der `notifications/claude/channel`-Empfang **direkt** an Sub-Agent-Spawn koppelt.
- Kein Handler der per Channel-Event `Task`-Tool aufruft.
- Strings im 228 MB Binary brachten 0 Treffer fuer `warmup|sidechain|channel.*spawn` (Bun-Bundle, Strings teils kompiliert/komprimiert — kein Beweis fuer Abwesenheit).

**Konsequenz fuer die Theorie:**
- **Direkt-Kopplung Channel → Warmup-Sub-Agent: nicht belegbar.** Channels-Doku und Code stimmen ueberein: Tag-Injection in Parent-Context, kein Auto-Spawn.
- **`warmup_minimal` als Goal-Category existiert** — Sub-Agents werden also irgendwo mit dieser Kategorie gestartet. Wahrscheinliche Quelle: ein **Anthropic-internes Cache-Warming** beim Session-Resume oder Plugin-Sync (separate Mechanik, nicht Channel-getriggert) ODER der **claude-mem-Worker** der periodisch eigene `agent-*.jsonl`-Summary-Tasks startet (siehe Befund vom Vormittag: `e68a5981...jsonl` mit claude-mem-Boilerplate, das ist genau dieser Mechanismus).
- **Marathon-Skill** war heute hier nicht aktiv (Marker fehlt, keine Cron-Jobs).

**Schlussfolgerung:** Die User-Theorie hat ein wahres Kern-Phaenomen (Sub-Agents mit `warmup_minimal` existieren, `agent-*.jsonl` ist real, kostet Tokens) — aber der **Trigger ist nicht das Channels-Feature**, sondern Hintergrund-Worker wie **claude-mem** die Summary-Sub-Agents pro Session generieren. Bridge und Channels sind clean.

**Empfehlung:** claude-mem-Verhalten checken (wie oft startet er pro Stunde Sub-Agents? welcher Model?). Optional: claude-mem-Worker auf Haiku zwingen via Modell-Lock oder Frequenz drosseln.
